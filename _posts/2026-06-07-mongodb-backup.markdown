# How to Back Up and Restore MongoDB

Backing up your database is one of the most critical tasks in database administration. In MongoDB, the standard tools for creating logical backups and restoring them are **mongodump** and **mongorestore**. 

These utility tools export database data into binary dump files (BSON/JSON format) and import them back, making them fast, space-efficient, and easy to transfer.

In this guide, we will cover how to use both tools with practical command examples.

---

## 1. Installation

`mongodump` and `mongorestore` are part of the **MongoDB Database Tools** package. If they are not already installed on your system, you can download them from the [MongoDB Download Center](https://www.mongodb.com/try/download/database-tools).

---

## 2. Using `mongodump`

The `mongodump` utility connects to a running MongoDB instance and creates a BSON representation of the database.

### Basic Syntax
```bash
mongodump [options]
```

### Common Command Examples

#### A. Backup All Databases (Local Instance)
If MongoDB is running locally on the default port (`27017`) and doesn't require authentication, simply run:
```bash
mongodump
```
This command creates a folder named `dump/` in your current working directory, with separate subdirectories for each database.

#### B. Backup a Specific Database
To target a specific database (e.g., `my_database`) and write the output to a custom directory:
```bash
mongodump --db my_database --out /path/to/backup/
```

#### C. Backup a Specific Collection
If you only need to back up a single collection (e.g., `users` inside `my_database`):
```bash
mongodump --db my_database --collection users --out /path/to/backup/
```

#### D. Compress and Archive (Recommended)
By default, `mongodump` creates individual BSON/JSON files for each collection. To compress your backup into a single archive file, use the `--archive` and `--gzip` options:
```bash
mongodump --db my_database --archive=my_database_backup.gz --gzip
```

#### E. Backup from a Remote MongoDB Instance / MongoDB Atlas
To connect to a remote instance or a MongoDB Atlas cluster, use the connection URI string:
```bash
mongodump --uri="mongodb+srv://username:password@cluster.example.com/my_database" --out /path/to/backup/
```

## 3. Using `mongorestore`

The `mongorestore` utility recreates database collections and indexes from the binary backups generated by `mongodump`.

### Basic Syntax
```bash
mongorestore [options] [path to backup]
```

### Common Command Examples

#### A. Restore All Databases from a Directory
To restore all databases located in your `dump/` folder:
```bash
mongorestore dump/
```

#### B. Restore a Specific Database
To restore a single database from its corresponding backup directory:
```bash
mongorestore --db my_database dump/my_database/
```

#### C. Restore from a Compressed Archive
If you created a compressed archive using `--archive` and `--gzip`, restore it with:
```bash
mongorestore --archive=my_database_backup.gz --gzip
```

#### D. Drop Existing Collections Before Restoring (`--drop`)
By default, `mongorestore` inserts records and merges data into existing collections. If you want to overwrite the existing collections completely, use the `--drop` option. This drops each collection from the target database before restoring it:
```bash
mongorestore --db my_database --drop dump/my_database/
```
> [!WARNING]
> Using `--drop` will delete existing data in the collections being restored. Use this option with caution!

#### E. Rename Database During Restore
If you want to restore data from a backup of `my_database` into a new database named `my_database_dev`, you can use `--nsFrom` and `--nsTo` to rename the namespace:
```bash
mongorestore --nsFrom="my_database.*" --nsTo="my_database_dev.*" dump/my_database/
```

## 4. Best Practices

1. **Avoid Production Overhead:** Running `mongodump` can impact database performance because it reads all documents. For large production databases, run backups against a secondary replica set member to minimize load on the primary node.
2. **Automate with Cron Jobs:** Set up scheduled cron jobs to automatically back up your database, compress the output, and upload the archives to a secure cloud storage solution (like AWS S3 or Google Cloud Storage).
3. **Verify Restores Periodically:** A backup is only as good as its restore process. Regularly test your backups by restoring them to a staging or development environment to verify data integrity.
